## 一、成本核算方法

### 1. 移动加权平均法 (主推)

每次买入后即时重算均价, 减仓用当前均价结转成本。公式 (Wikipedia 移动平均成本):

```
新均价 = (原有数量×原均价 + 新买数量×新买价) / (原有数量 + 新买数量)
```

**示例 (恒瑞 600276)**: 假设先 100 股 @49.121, 再加 100 股 @50.00:
```
新均价 = (100×49.121 + 100×50.00) / 200 = 49.5605
```

### 2. 先进先出法 (FIFO)

最早买入的批次最先结转。多批次时 COGS 用最早层成本, 剩余持仓是最近批次。

### 3. 加仓后均价变化

加仓**必然**把均价拉向新买价: 高于均价加仓→均价上行; 低于均价加仓→均价下行。这是数学铁律, 不存在 "加仓后成本不变"。

### 4. 部分减仓后剩余成本 — 本节核心修正点

**铁律: 部分减仓不改变剩余持仓的成本价, 只减少数量。** 减仓结转的是 "已实现盈亏", 剩余 lot 的成本基原封不动。

本项目 `decisions` 表用 **lot 制 (specific identification)** 实现: 每笔 `reduce` 行通过 `parent_id` 挂到具体 `buy` 行, `reduce_position()` 按 `(卖出价 − 父lot买入价) × 减仓量` 算 pnl, 并立即给 reduce 行打 `close_date`。父 buy 行本身**不**被关闭, 剩余数量 = `buy.qty − Σreduce.qty`, 成本仍是父 buy 的原价。

**实测 (300059 东财)**:

| id | action | date | price | qty | parent_id | 含义 |
|---|---|---|---|---|---|---|
| 395 | buy | 06-26 | **21.279** | 300 | — | 真实成本 21.279 |
| 1200 | reduce | 06-29 | 20.05 | 200 | 395 | 卖出价 20.05, pnl=−5.78% |

减仓 200 股后, 剩余 100 股成本**仍是 21.279**, 不是 20.05, 也不是任何 "新均价"。已实现亏损 = (20.05−21.279)×200 = **−245.8 元**。

### 5. 为什么 "瞎猜成本" 是致命错误

成本是盈亏核算、止损位、加仓决策、税基的**唯一锚**。猜错成本会级联放大:

1. **盈亏失真** → 以为亏的是 "脏数据", 实际是真亏, 风险被掩盖。
2. **决策反转** → 据错误成本判断 "不该减仓/该加仓", 方向全反。
3. **止损失灵** → 真成本 21.279 vs 猜 20.07, 止损线差 6%, 真破位时还在硬扛。
4. **复盘污染** → 归因表全是错的, 越复越偏。

**CFO 今天错在哪**: 把 `reduce` 行的**卖出价 20.05** (PROJECT_STATE 误写成 20.07) 当成了**持仓成本**, 反咬 "db 虚亏有脏数据"。实际 db 完全正确, 21.279 才是真成本。本质是**没查 db 就报数**。

## 二、盈亏核算

### 1. 已实现盈亏 vs 浮动盈亏

- **已实现盈亏 (Realized PnL)** = 已平仓部分的 `(卖出价 − 成本) × 数量`。落袋, 不可逆。对应 `decisions` 表里 `action='reduce'` 且 `close_date` 非空的行。
- **浮动盈亏 (Unrealized PnL)** = 剩余持仓 `(现价 − 成本) × 剩余数量`。随行情波动, 未实现。成本取父 lot 买入价。

### 2. 精确计算法 (从 decisions 表)

```sql
-- 已实现: 每笔 reduce 挂回 parent buy 成本
SELECT r.code, (r.price - b.price) * r.quantity AS realized_pnl
FROM decisions r JOIN decisions b ON r.parent_id = b.id
WHERE r.action='reduce' AND r.decision_date='2026-06-29';

-- 浮动: 剩余 lot × (现价 − 成本), 剩余量 = b.qty − Σ同parent的reduce.qty
```

## 三、仓位管理

### 1. 凯利公式 (Kelly Criterion, 已查证 Wikipedia)

二值下注: `f* = p − q/b`, 其中 p=胜率, q=1−p, b=赔率(赢的倍数)。
投资一般式 (部分亏损): `f* = p/l − q/g`, g=赢的涨幅, l=亏的跌幅。

**分数凯利 (Fractional Kelly)**: 实战用半凯利 / 四分之一凯利。理由 (Wikipedia 原文): 全额凯利回撤大、对概率估计误差敏感, "betting larger than Kelly increases risk of ruin"。半凯利牺牲少量增长率换大幅降低破产概率与波动。

### 2. 固定比例 / 风险预算 (VaR 约束)

- **固定比例**: 单仓不超总资产 X% (本项目 rules.yaml 用 `plan_max_position_pct`)。
- **风险预算**: 用 VaR 反推仓位。本项目 `risk_metrics.compute()` 算 `var_95_1d = 1.645 × 日波动 × 总资产`, 阈值 >3% 触发提示。仓位上限 = 可承受最大亏损 ÷ 单标的日波动。

### 3. 集中度上限

- **单一标的**: 个人经验上限 20–30%, 本项目 `risk_metrics` 触发线 `largest_pct > 30%`。
- **单一板块**: 上限 40–50%, 触发线 `max_pct > 50%`。
- **持仓只数**: <4 只提示分散不足。

### 4. 现金仓位的作用

现金双重职能: **弹药**(机会出现时能买) + **缓冲**(跌时不被迫割, 心理稳定)。今天现金从 ~46% 升到 **69%**, 弹药极充足, 但代价是**进攻力下降** — 目标 185 天 +26.9%, 69% 现金意味着剩余 31% 仓位要扛起全部收益, 年化要求被放大 ~3 倍。蒙特卡洛基线 P(达成)=0.2%, 高现金进一步压低概率。得失: 防御满分, 进攻不足, 需在回调中果断回补。

## 四、盈亏归因

### 1. 三维归因

- **按标的**: 谁赚谁亏。今日恒瑞 +354.9 (主功), 东财 −245.8 (拖累), 创业板 −119 (操作失误), 芯片 +77.2 (执行尚可)。
- **按板块**: 消费/医药/科技/金融分别贡献。今日医药贡献正, 金融负。
- **按策略**: mid vs short, 止盈 vs 止损。今日 4 笔标记 `partial_take_profit`, 但创业板/东财实际是亏损减仓, **标签与实质不符** — 这是归因失真的隐患。

### 2. Brinson 模型 (概念)

把组合超额收益拆为 **配置效应 (Allocation)** + **选股效应 (Selection)** + **交互效应 (Interaction)**。本项目数据粒度暂不足以跑完整 Brinson, 但思路适用: 区分 "赚在选股" 还是 "赚在仓位轻重"。

### 3. 今日归因结论

赚钱来源 = 恒瑞中线择股 + 持有耐心 (买 49.121 → 52.67, +7.2%)。
亏钱来源 = ① 东财买点偏高 (21.279 接近阶段高), 趋势未破却减在亏位; ② 创业板盘中急杀浅止损割在低点, 下午 V 回踏空。**操作问题 > 选股问题**。

## 七、项目落地建议

### 1. `risk_metrics.py` 增强

- **加成本基**: `_load_positions()` 目前只算数量和市值, **不算成本**。建议新增 `cost_basis` 字段 = 父 lot 买入价 (lot 制) 或移动加权均价, 并输出 `unrealized_pnl`。这样风险报告自带盈亏, 不依赖外部口算。
- **双口径集中度**: 现有 `sector_concentration` 是股票内部分布。加一个 "含现金口径" 选项, 两者并报, 避免高现金掩盖股票集中度。
- **reduce 标签校验**: 归因前检查 `close_reason` 与 `pnl_pct` 符号一致 — `partial_take_profit` 但 pnl<0 → 标记异常。

### 2. `log.py` / `decision_log.py` 成本校验机制 (防再瞎猜)

- **报盈亏前强制查 db**: 新增 `cmd_cost` 子命令, 一键输出某 code 的真实成本:
  ```bash
  .venv/bin/python -m a_stock.log cost 300059
  # → 买 21.279×300 (id=395) | 已减 200@20.05 (realized −245.8) | 剩余 100 成本 21.279
  ```

- **PROJECT_STATE.md 成本列加校验**: 该文件 300059 行成本写成 20.07 (错), 是脏数据源头之一。建议写一个 `validate_state.py`: 对 PROJECT_STATE 每行成本, 反查 `decisions` 表父 lot 买入价, 不一致就报警。从源头堵住 "文档成本 ≠ db 成本"。
